# 자주 묻는 질문 (FAQ) - 소프트웨어 테스팅

## Q1. 테스팅이란 무엇인가요?

**A:** 테스팅은 소프트웨어가 요구사항을 충족하는지 확인하고 결함을 식별하기 위한 체계적인 과정입니다. 이는 소프트웨어 품질 보장의 핵심 활동으로, 사용자에게 신뢰성 있는 제품을 제공하는 데 중요한 역할을 합니다.

## Q2. 테스트의 주요 목표는 무엇인가요?

**A:** 테스트의 주요 목표는 다음과 같습니다:
- 결함 탐지 및 제거
- 품질 보증
- 위험 감소
- 요구사항 검증
- 시스템 신뢰성 향상

## Q3. 블랙박스 테스트와 화이트박스 테스트의 차이점은 무엇인가요?

**A:**
- **블랙박스 테스트**: 내부 구조를 알지 못한 채 입력과 출력만을 기반으로 테스트하는 방식입니다. 주로 기능 테스트에 사용됩니다.
- **화이트박스 테스트**: 내부 구조와 로직을 기반으로 테스트하는 방식입니다. 코드 커버리지 등을 검증할 때 사용됩니다.

## Q4. 결정 테이블(Decision Table)이란 무엇인가요?

**A:** 결정 테이블은 다양한 조건 조합에 따른 시스템의 동작을 표 형식으로 표현한 테스트 기법입니다. 조건(Conditions)과 결과(Actions)로 구성되어 복잡한 비즈니스 로직을 시각적으로 표현할 수 있습니다. 자세한 내용은 [결정 테이블 상세 설명](#decision_table)을 참조하세요.

## Q5. 결정 테이블 작성 4단계는 무엇인가요?

**A:** 결정 테이블 작성은 다음과 같은 4단계로 구성됩니다:

1. **조건(Conditions)과 결과(Actions) 식별**: 시스템의 입력 조건과 그에 따른 출력 행동을 모두 나열합니다.
2. **규칙(Rules)의 개수 계산**: 가능한 규칙의 총 개수는 $2^n$ ($n$ = 조건의 개수)입니다.
3. **테이블 구성 (조합 채우기)**: 조건의 조합을 빠짐없이 기입합니다.
4. **결과(Actions) 기입 및 최적화**: 각 조건 조합에 맞는 결과를 체크하고 규칙을 합칠 수 있습니다.

자세한 내용은 [결정 테이블 상세 설명](#decision_table)을 참조하세요.

## Q6. 테스트 자동화를 언제 도입해야 하나요?

**A:** 테스트 자동화는 다음과 같은 경우에 도입하는 것이 좋습니다:
- 반복적인 회귀 테스트가 필요한 경우
- 수동 테스트가 시간이 많이 소요되는 경우
- 테스트 커버리지를 높이고 싶은 경우
- CI/CD 파이프라인에 통합하고자 할 때

## Q7. 단위 테스트, 통합 테스트, 시스템 테스트의 차이점은 무엇인가요?

**A:**
- **단위 테스트**: 개별 구성 요소나 함수 수준에서의 테스트 (예: 개별 함수, 클래스)
- **통합 테스트**: 여러 구성 요소 간 상호작용을 검증하는 테스트
- **시스템 테스트**: 전체 시스템의 기능 및 비기능 요구사항을 검증하는 테스트

## Q8. 테스트 커버리지란 무엇이며 왜 중요한가요?

**A:** 테스트 커버리지는 테스트가 소프트웨어의 어느 정도를 커버하는지를 나타내는 지표입니다. 예를 들어, 구문 커버리지는 코드 문장 중 얼마나 많은 비율이 테스트되었는지를 보여줍니다. 높은 커버리지는 결함 발견 가능성을 높이지만, 결함이 없는 것을 보장하지는 않습니다.

## Q9. V-모델과 애자일 테스트의 차이점은 무엇인가요?

**A:**
- **V-모델**: 개발과 테스트가 대칭적인 관계를 이루며, 각 개발 단계에 대응하는 테스트 단계가 있습니다. 문서 중심의 체계적 접근이 특징입니다.
- **애자일 테스트**: 반복적이고 점진적인 방식으로 테스트가 수행되며, 고객 피드백을 빠르게 반영합니다. 유연성과 협업이 강조됩니다.

## Q10. 탐색적 테스트란 무엇인가요?

**A:** 탐색적 테스트는 테스트 설계와 실행을 동시에 수행하는 테스트 기법입니다. 테스터의 경험과 직관을 바탕으로 실시간으로 테스트를 설계하고 실행하며, 창의적인 테스트가 가능합니다.

## Q11. 결함의 심각도와 우선순위의 차이점은 무엇인가요?

**A:**
- **심각도**: 결함이 시스템에 미치는 영향의 정도 (Critical, High, Medium, Low)
- **우선순위**: 결함을 수정할 순서 (Immediate, High, Medium, Low)
심각한 결함이라도 수정 우선순위는 낮을 수 있습니다.

## Q12. 테스트 데이터 관리는 왜 중요한가요?

**A:** 테스트 데이터 관리는 다음과 같은 이유로 중요합니다:
- 일관된 테스트 결과 보장
- 테스트 재현성 확보
- 개인정보 보호 및 보안 유지
- 테스트 실행 속도 향상

## Q13. 성능 테스트에는 어떤 종류가 있나요?

**A:** 성능 테스트의 주요 유형은 다음과 같습니다:
- **로드 테스트**: 정상적인 사용자 수에서 시스템 성능 확인
- **스트레스 테스트**: 최대 용량 이상의 부하에서 시스템 반응 확인
- **볼륨 테스트**: 대량 데이터 처리 능력 확인
- **스테이빌리티 테스트**: 장시간 운영 시 안정성 확인

## Q14. 테스트 환경 관리의 중요성은 무엇인가요?

**A:** 테스트 환경은 테스트 결과의 신뢰성에 직접적인 영향을 미치므로 다음과 같은 이유로 중요합니다:
- 일관된 테스트 결과 제공
- 테스트 재현성 보장
- 테스트 실행 속도 향상
- 환경 간 차이로 인한 문제 예방

## Q15. 테스트 자동화 프레임워크를 선택할 때 고려해야 할 사항은 무엇인가요?

**A:** 테스트 자동화 프레임워크 선택 시 다음을 고려해야 합니다:
- 프로젝트 기술 스택과의 호환성
- 팀의 기술 역량
- 유지보수 용이성
- 확장성
- 커뮤니티 지원

## Q16. 회귀 테스트의 중요성은 무엇인가요?

**A:** 회귀 테스트는 코드 변경으로 인해 기존 기능에 문제가 발생하지 않는지 확인하는 테스트입니다. 소프트웨어 변경 시 기존 기능이 여전히 제대로 작동하는지 보장하는 데 매우 중요합니다.

## Q17. 테스트 관리 도구를 사용하는 이유는 무엇인가요?

**A:** 테스트 관리 도구는 다음과 같은 이유로 사용됩니다:
- 테스트 케이스 체계적 관리
- 테스트 실행 추적
- 결함 관리 및 추적
- 팀 협업 향상
- 테스트 보고서 자동 생성
