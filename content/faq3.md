# 자주 묻는 질문 (FAQ) - 소프트웨어 테스팅 (3)

## Q15. 테스트 자동화 프레임워크를 선택할 때 고려해야 할 사항은 무엇인가요?

**A:** 테스트 자동화 프레임워크 선택 시 다음을 고려해야 합니다:
- 프로젝트 기술 스택과의 호환성
- 팀의 기술 역량
- 유지보수 용이성
- 확장성
- 커뮤니티 지원

## Q16. 회귀 테스트의 중요성은 무엇인가요?

**A:** 회귀 테스트는 코드 변경으로 인해 기존 기능에 문제가 발생하지 않는지 확인하는 테스트입니다. 소프트웨어 변경 시 기존 기능이 여전히 제대로 작동하는지 보장하는 데 매우 중요합니다.

## Q17. 테스트 관리 도구를 사용하는 이유는 무엇인가요?

**A:** 테스트 관리 도구는 다음과 같은 이유로 사용됩니다:
- 테스트 케이스 체계적 관리
- 테스트 실행 추적
- 결함 관리 및 추적
- 팀 협업 향상
- 테스트 보고서 자동 생성

## Q18. 결함 생명 주기 (Defect Life Cycle)란 무엇인가요?

**A:** 결함 생명 주기는 테스터가 결함을 발견했을 때부터 수정이 완료되어 종료될 때까지의 과정입니다. 시험에서는 각 상태(Status)의 의미를 묻는 문제가 단골로 나옵니다.
- **New (신규)**: 테스터가 결함을 새로 등록한 상태.
- **Assigned (할당)**: 결함을 수정할 개발자에게 배정된 상태.
- **Resolved/Fixed (수정 완료)**: 개발자가 코드를 수정한 상태.
- **Re-test (재테스트)**: 테스터가 수정이 잘 되었는지 다시 확인하는 과정.
- **Closed (종료)**: 결함이 완료히 해결되었음을 확인하고 마무리.
- **Deferred (연기)**: 중요도가 낮아 이번 버전에서는 수정하지 않기로 결정한 상태.

## Q19. 테스트 레벨별 실무 (Testing Levels)의 차이점은 무엇인가요?

**A:** 실무에서는 개발 단계에 따라 테스트의 목적과 주체가 명확히 나뉩니다.

| 레벨 | 주요 대상 | 수행 주체 | 실무 포인트 |
|------|-----------|-----------|-------------|
| **단위 테스트** | 컴포넌트, 모듈 | 개발자 | 코드가 의도한 대로 동작하는지(함수 단위) 확인 |
| **통합 테스트** | 모듈 간 인터페이스 | 개발자/테스터 | 모듈을 합쳤을 때 데이터가 잘 주고받아지는지 확인 |
| **시스템 테스트** | 전체 소프트웨어 | 테스터 | 비기능 요소(성능, 보안)를 포함한 전체 요구사항 확인 |
| **인수 테스트** | 완제품 | 고객/사용자 | 실제 비즈니스 환경에서 쓸 수 있는지 최종 승인 |

## Q20. 리스크 기반 테스트 (Risk-based Testing)란 무엇인가요?

**A:** 리스크 기반 테스트는 현업에서는 시간과 자원이 한정되어 있어 모든 것을 테스트할 수 없을 때 사용하는 실무 전략입니다.
- **리스크 산정**: **장애 발생 가능성(Likelihood)**과 **비즈니스 영향도(Impact)**를 곱해서 우선순위를 정합니다.
- **전략**:
  - 리스크가 높은 기능은 경험 많은 테스터를 배치하고 **강도 높은 기법(화이트박스)**을 적용합니다.
  - 리스크가 낮은 기능은 나중에 테스트하거나 간소화합니다.

## Q21. 테스트 자동화 (Test Automation) 시 고려사항은 무엇인가요?

**A:** 테스트 자동화 도구 도입 시 다음을 고려해야 합니다:
- **회귀 테스트(Regression Test)**: 수정 후 기존 기능이 망가지지 않았는지 반복 확인해야 하므로 자동화의 가장 큰 타겟이 됩니다.
- **주의점**: 모든 테스트를 자동화할 수는 없습니다. 도구 구입비, 스크립트 유지보수 비용 등 **비용 대비 효율(ROI)**을 반드시 따져야 합니다.
