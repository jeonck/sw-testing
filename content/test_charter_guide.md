# '테스트 차터' 작성법: 90분 안에 서비스의 약점을 털어내는 전략

탐색적 테스팅이 '무작정 누르기'가 되지 않게 만드는 핵심 장치가 바로 **테스트 차터(Test Charter)**입니다. 차터는 테스터에게 목적지를 알려주는 나침반과 같습니다.

짧은 시간 안에 서비스의 핵심 약점을 공략하기 위한 실무형 테스트 차터 작성법을 정리해 드립니다.

## 📂 [테스트 수행] 테스트 차터 작성 가이드

### 1. 테스트 차터(Test Charter)란?

테스트 차터는 "무엇을(Target), 어떤 관점에서(Focus), 어떤 도구를 사용하여(Resources) 탐색할 것인가"를 한 문장으로 정의한 지침서입니다.

- **핵심 공식**: [대상]을 [자원/기구]를 활용하여 [목표/관점]에 대해 탐색한다.

- **예시**: "결제 시스템을 포인트/쿠폰 복합 결제 시나리오를 활용하여 데이터 정합성 관점에서 탐색한다."

### 2. 90분 집중 전략 (Session-Based Testing)

탐색적 테스팅은 무제한으로 수행하는 것이 아니라, 60~90분 정도의 집중된 세션으로 진행할 때 가장 효율적입니다.

- **준비 (10분)**: 차터 작성 및 테스트 데이터(계정, 포인트 등) 세팅.

- **수행 (70분)**: 차터에 집중하여 탐색 수행. 발견된 현상은 즉시 메모하고, 의심되는 지점은 깊게 파고들기.

- **마무리 (10분)**: 발견된 결함(Bug) 등록 및 테스트 노트 정리.

### 3. 차터 작성을 위한 5가지 핵심 요소 (PROBE)

실무에서 차터를 쓸 때 다음 5가지를 포함하면 내용이 풍성해집니다.

- **P (Purpose)**: 이번 탐색의 목적 (예: 신규 기능의 안정성 확인)

- **R (Resources)**: 필요한 도구 (예: Charles Proxy, 특정 단말기, 관리자 페이지)

- **O (Objects)**: 테스트 대상 (예: 검색 필터 UI, 회원가입 API)

- **B (Boundaries)**: 제외 범위 (예: 결제 모듈은 제외, 단순 UI 레이아웃은 무시)

- **E (Examine)**: 검증할 가설 (예: "검색어에 특수문자를 넣으면 결과값이 깨질 것이다")

### 4. 실무형 테스트 차터 양식 및 예시

테스터들이 현업에서 바로 사용할 수 있는 간단한 양식입니다.

| 항목 | 작성 예시 |
|------|-----------|
| **차터명** | 장바구니 및 주문서 데이터 동기화 탐색 |
| **테스트 대상** | 장바구니 담기 ~ 주문서 진입 단계 |
| **탐색 목표** | 멀티 디바이스 환경에서 수량 변경 시 주문서에 즉시 반영되는가? |
| **사용 도구** | PC(Chrome), Mobile(iOS), 관리자 DB 조회 쿼리 |
| **가설(Tactics)** | "PC에서 담은 상품 수량을 모바일 앱 주문서 진입 직전에 수정하면, 최종 결제 금액에 오류가 발생할 것이다." |

### 5. 차터 작성 시 주의할 점 (Anti-Patterns)

- **너무 넓은 범위**: "전체 기능 테스트" (X) → "로그인 세션 만료 시의 예외 처리 탐색" (O)

- **너무 뻔한 내용**: "정상적으로 결제가 되는지 확인" (X) → "결제 도중 네트워크를 차단했을 때의 데이터 복구 확인" (O)

- **기록 없는 탐색**: 탐색 후에 "무엇을 했는지" 남지 않는다면 그것은 몽키 테스팅에 불과합니다. 반드시 테스트 노트를 병행하세요.

## 📝 테스터를 위한 한 줄 전략

"차터는 테스터를 가두는 감옥이 아니라, 가장 위험한 곳으로 안내하는 지도입니다." 정해진 시나리오가 지루하게 느껴질 때, 스스로 차터를 작성해 서비스의 허점을 찔러보세요.
