# 수정된 버그만 보지 마세요: 영향도 분석을 통한 회귀 테스트 범위 산정법

버그가 수정되었다는 소식을 들었을 때, 테스터가 가장 먼저 하는 실수는 **'고쳐진 그 지점'**만 확인하고 테스트를 끝내는 것입니다. 하지만 소프트웨어는 복잡하게 얽혀 있어, 한 곳을 고치면 전혀 예상치 못한 곳에서 새로운 버그가 터지곤 합니다.

이것이 바로 **회귀 테스트(Regression Test)**가 필요한 이유입니다. 한정된 시간 내에 효율적으로 범위를 잡는 영향도 분석 기반의 회귀 테스트 전략을 정리해 드립니다.

## 📂 [테스트 수행] 영향도 분석을 통한 회귀 테스트 범위 산정법

### 1. 왜 영향도 분석(Impact Analysis)이 중요한가?

모든 기능을 다시 테스트하는 '전수 회귀 테스트'는 현실적으로 불가능합니다.

- **리스크 방지**: 수정 사항이 기존의 정상적인 기능(Legacy)을 망가뜨렸는지 확인해야 합니다.

- **리소스 최적화**: 영향이 거의 없는 곳은 제외하고, 위험한 곳에 화력을 집중합니다.

### 2. 영향도 분석을 위한 3단계 접근법

#### ① 1단계: 직접 영향 범위 (Direct Impact)

수정된 코드나 모듈이 직접적으로 포함된 기능입니다.

- **예**: '로그인 버튼 색상 변경' → 로그인 기능 자체, 로그인 팝업 내 텍스트 등.

- **테스트 수준**: 고쳐진 기능의 모든 케이스를 다시 확인합니다.

#### ② 2단계: 간접 영향 범위 (Indirect Impact)

데이터를 공유하거나 상하 관계에 있는 기능들입니다.

- **예**: '장바구니 담기 로직 수정' → 장바구니 리스트 조회, 결제 금액 계산, 마이페이지 최근 본 상품 등.

- **테스트 수준**: 주요 시나리오(Happy Path) 위주로 확인합니다.

#### ③ 3단계: 공통 모듈 및 시스템 전체 (System-wide Impact)

로그인, 검색, 결제 등 서비스 전체에 영향을 주는 공통 컴포넌트나 라이브러리 수정 시 해당합니다.

- **예**: '서버 공통 응답 포맷 변경' 또는 'GNB(상단 메뉴) 레이아웃 수정'.

- **테스트 수준**: 서비스의 핵심 기능(Critical Path) 전체를 훑는 스모크 테스트를 수행합니다.

### 3. 실무적인 범위 산정 기준 (RTM 활용)

요구사항 추적표(RTM)나 모듈 구조도가 있다면 훨씬 정교하게 범위를 짤 수 있습니다.

| 수정 유형 | 권장 회귀 테스트 범위 | 비고 |
|-----------|------------------------|------|
| **단순 UI/문구 수정** | 해당 화면 및 연결된 링크 확인 | 사이드 이펙트 낮음 |
| **DB 스키마 변경** | 해당 데이터를 사용하는 모든 CRUD 기능 | 데이터 유실/왜곡 위험 높음 |
| **API 인터페이스 변경** | 해당 API를 호출하는 프론트엔드 전 영역 | 통신 오류 발생 가능성 높음 |
| **환경 설정/라이브러리 업데이트** | 서비스 전체 스모크 테스트 + 주요 비즈니스 로직 | 시스템 불안정성 극대화 |

### 4. 회귀 테스트의 우선순위 설정 기법 (RBT)

시간이 정말 부족하다면 **'실패했을 때 가장 치명적인 곳'**부터 테스트합니다.

- **P0 (Critical)**: 결제, 로그인, 회원가입 등 서비스 중단과 직결되는 기능.

- **P1 (High)**: 사용자가 매일 쓰는 핵심 기능 (예: 상품 검색, 게시글 작성).

- **P2 (Normal)**: 사용 빈도는 낮지만 비즈니스상 필요한 기능 (예: 프로필 수정, 공지사항 조회).

- **P3 (Low)**: 단순 정보 노출 페이지 (예: 회사 소개, 이용약관).

### 5. 실무 보고 팁: "왜 여기만 테스트했나요?"라는 질문에 답하기

감리사나 기획자가 테스트 범위에 대해 물을 때, 논리적으로 답변하는 서술 방식입니다.

#### [회귀 테스트 설계 근거]

"금번 수정 사항은 **'주소지 입력 로직'**의 변경으로, 해당 데이터를 직접 참조하는 **'배송비 계산 모듈'**과 **'최종 결제 페이지'**를 2차 영향 범위로 산정했습니다. 상대적으로 영향도가 낮은 '마이페이지 주소록 관리'는 샘플링 테스트로 대체하여 효율성을 높였습니다."
