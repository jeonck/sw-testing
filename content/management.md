# 테스트 관리 (Test Management)

## 테스트 조직

### 테스트 조직의 독립성

테스트 조직의 독립성은 테스트의 객관성과 효과성을 보장하는 핵심 요소입니다.

#### 독립성의 수준

1. **개발 팀 내 테스터**
   - 장점: 빠른 피드백, 커뮤니케이션 용이
   - 단점: 객관성 부족, 개발자와의 갈등

2. **개발 팀과 분리된 독립 테스트 팀**
   - 장점: 객관적 테스트, 전문성 강화
   - 단점: 커뮤니케이션 지연, 책임 감소

3. **독립된 테스트 부서**
   - 장점: 높은 전문성, 일관된 품질 기준
   - 단점: 비용 증가, 프로젝트 적응 시간 필요

4. **외부 테스트 조직**
   - 장점: 높은 독립성, 전문 지식
   - 단점: 도메인 지식 부족, 보안 문제

#### 독립성의 이점

- **객관성**: 개발자의 시각에서 벗어난 테스트
- **전문성**: 테스트 전용 역량 집중
- **효율성**: 테스트 자동화 및 도구 전문화
- **리스크 관리**: 독립적인 품질 평가

### 테스트 조직의 역할과 책임

#### 테스트 매니저

- 테스트 전략 수립
- 테스트 계획 및 일정 관리
- 팀원 관리 및 배치
- 상위 관리자와의 커뮤니케이션
- 품질 지표 분석 및 보고

#### 테스트 리더/시니어 테스터

- 테스트 설계 및 검토
- 테스트 자동화 개발
- 테스트 팀 기술 향상
- 복잡한 테스트 시나리오 수행

#### 테스터

- 테스트 케이스 작성
- 테스트 실행 및 결과 기록
- 결함 보고 및 추적
- 테스트 데이터 준비

## 테스트 계획 및 추정

### 테스트 계획 수립

#### 테스트 계획의 구성 요소

1. **테스트 범위 (Test Scope)**
   - 테스트 대상 시스템
   - 테스트 제외 항목
   - 기능 및 비기능 테스트 범위

2. **테스트 접근 방식 (Test Approach)**
   - 테스트 유형 및 전략
   - 테스트 기법 선택
   - 자동화 전략

3. **테스트 항목 (Test Items)**
   - 테스트 대상 구성 요소
   - 시스템 인터페이스
   - 외부 시스템

4. **테스트 완료 기준 (Exit Criteria)**
   - 결함 밀도 기준
   - 테스트 커버리지 기준
   - 위험 수용 기준

5. **위험 및 대응 방안**
   - 프로젝트 위험 식별
   - 위험 대응 전략
   - 예비 대응 계획

#### 테스트 전략 수립

- **위험 기반 테스트**: 높은 위험 항목에 우선순위 부여
- **요구사항 기반 테스트**: 요구사항 충족 여부 검증
- **모델 기반 테스트**: 시스템 모델을 기반으로 테스트 생성

### 테스트 일정 및 자원 추정

#### 일정 추정 기법

1. **Expert Judgment**
   - 경험 기반 추정
   - 팀원 참여 및 검토

2. **Analogous Estimation**
   - 유사 프로젝트 경험 기반
   - 과거 데이터 활용

3. **Three-Point Estimation**
   - 낙관적/비관적/가장 가능 시간 평균
   - (O + 4M + P) / 6

#### 자원 추정 요소

- 테스터 수 및 기술 수준
- 테스트 환경 및 도구
- 테스트 데이터 필요량
- 테스트 자동화 요구사항

   - 도구 재설정

## 결함 관리

### 결함 생명 주기 (Life Cycle)

#### 결함 상태

1. **New (신규)**: 결함이 처음 보고됨
2. **Open (개방)**: 테스트 팀이 검토 중
3. **Assigned (할당)**: 개발자에게 할당됨
4. **Fixed (수정)**: 개발자가 수정 완료
5. **Retest (재테스트)**: 테스트 팀이 검증 중
6. **Verified (검증)**: 수정 확인 완료
7. **Closed (폐기)**: 결함 처리 완료
8. **Rejected (거부)**: 결함이 아닌 것으로 판정
9. **Deferred (연기)**: 다음 버전으로 연기

#### 결함 처리 프로세스

1. **결함 식별**
   - 테스트 실행 중 결함 발견
   - 예상 결과와 실제 결과 비교

2. **결함 보고**
   - 결함 재현 단계 상세 기록
   - 결함 심각도 및 우선순위 지정
   - 관련 증거 자료 첨부

3. **결함 분석**
   - 결함 원인 분석
   - 영향 범위 파악
   - 수정 방법 검토

4. **결함 수정**
   - 코드 수정
   - 테스트 환경 반영
   - 관련 테스트 수행

5. **결함 검증**
   - 수정 사항 재테스트
   - 회귀 테스트 수행
   - 결함 상태 업데이트

### 결함 보고서 작성

#### 결감 보고서 구성 요소

1. **결함 식별 정보**
   - 결함 ID
   - 보고 날짜
   - 보고자

2. **결함 설명**
   - 결함 요약
   - 상세 설명
   - 재현 단계

3. **결함 영향**
   - 결함 심각도
   - 결함 우선순위
   - 영향 받는 모듈

4. **결함 증거**
   - 스크린샷
   - 로그 파일
   - 재현 조건

#### 결함 심각도 (Severity)

- **Critical**: 시스템 다운, 데이터 손실, 보안 위협
- **High**: 주요 기능 손상, 중요한 사용자 영향
- **Medium**: 부차적 기능 문제, 사용성 문제
- **Low**: 사소한 문제, UI 표시 문제

#### 결함 우선순위 (Priority)

- **Immediate**: 즉시 수정 필요
- **High**: 현재 릴리즈에서 수정 필요
- **Medium**: 다음 릴리즈에서 수정
- **Low**: 향후 고려

## 형상 관리

### 테스트 자산의 버전 관리

#### 형상 항목

1. **테스트 문서**
   - 테스트 계획
   - 테스트 케이스
   - 테스트 시나리오

2. **테스트 스크립트**
   - 수동 테스트 절차
   - 자동화 테스트 스크립트
   - 데이터 준비 스크립트

3. **테스트 데이터**
   - 테스트 입력 데이터
   - 기대 결과 데이터
   - 환경 설정 파일

4. **테스트 환경**
   - 환경 구성 정보
   - 설정 스크립트
   - 의존성 정보

#### 형상 관리 도구

- **Git**: 소스 코드 및 문서 형상 관리
- **SVN**: 중앙 집중형 형상 관리
- **Jenkins**: CI/CD 및 빌드 관리
- **JIRA**: 결함 및 이슈 관리
- **TestRail**: 테스트 케이스 관리

### 형상 관리 모범 사례

1. **일관된 네이밍 컨벤션**
   - 파일명 규칙
   - 브랜치 명명 규칙
   - 태그 명명 규칙

2. **버전 라벨링**
   - 릴리즈별 태그
   - 환경별 브랜치
   - 기능별 브랜치

3. **변경 추적**
   - 변경 이력 기록
   - 변경 사유 문서화
   - 검토 및 승인 절차

4. **백업 및 복구**
   - 정기 백업
   - 복구 절차 문서화
   - 백업 검증

### 테스트 관리 도구의 기능

1. **테스트 케이스 관리**
   - 케이스 작성 및 저장

   - 결함 보고 및 추적

   - 추세 분석

4. **통합 기능**
   - 형상 관리 도구 연동
