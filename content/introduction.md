# 소프트웨어 테스트의 기본 (Fundamentals of Testing)

## 테스트의 개요

소프트웨어 테스트는 소프트웨어 시스템이 의도한 대로 작동하는지 확인하고, 결함을 식별하며, 품질을 보장하기 위한 체계적인 과정입니다. 테스트는 소프트웨어 개발 생명 주기(SDLC)의 핵심 요소로, 제품의 신뢰성, 보안, 성능을 보장하는 데 중요한 역할을 합니다.

### 테스트의 정의

테스트는 다음과 같이 정의될 수 있습니다:

- **ISO/IEC 29119 표준**: "소프트웨어 기반 시스템, 모듈, 일부 또는 문서가 지정된 요구사항을 충족하는지, 또는 요구사항이 적절하고 완전한지 확인하기 위한 활동"
- **ISTQB 정의**: "분석 대상이 요구사항과 일치하는지 확인하고 결함을 식별하기 위한 조사 과정"

### 테스트의 필요성

- **품질 보장**: 사용자에게 고품질의 소프트웨어 제공
- **비용 절감**: 초기 단계에서 결함을 발견하면 수정 비용이 대폭 감소
- **신뢰성 확보**: 시스템이 안정적으로 작동함을 보장
- **보안 강화**: 보안 취약점 식별 및 대응
- **규제 준수**: 산업 표준 및 법적 요구사항 충족

### 테스트의 목표

1. **결함 탐지**: 코드, 설계, 구현에서 존재하는 결함 식별
2. **품질 보증**: 소프트웨어가 품질 기준을 충족하는지 확인
3. **위험 감소**: 출시 후 문제 발생 가능성 최소화
4. **요구사항 검증**: 소프트웨어가 사용자 요구사항과 명세서를 충족하는지 확인
5. **신뢰성 향상**: 시스템이 예상대로 작동함을 증명

## 테스트의 원리

소프트웨어 테스트에는 7가지 핵심 원칙이 존재합니다:

### 1. 테스팅은 결함이 존재함을 보여줄 뿐, 결함이 없음을 증명할 수 없다

- 테스트는 오직 결함의 존재를 보여줄 수 있을 뿐, 결함이 없음을 증명할 수는 없습니다.
- 테스트는 신뢰성을 높이지만, 완전한 신뢰를 보장할 수는 없습니다.

### 2. 살충제 패러독스 (Pesticide Paradox)

- 동일한 테스트를 반복적으로 수행하면 더 이상 새로운 결함을 찾지 못하게 됩니다.
- 정기적으로 테스트 케이스를 검토하고 수정하여 새로운 결함을 찾도록 해야 합니다.

### 3. 테스트는 요구사항에 따라 계획되어야 한다

- 모든 테스트는 명확한 요구사항과 기대 결과를 기반으로 계획되어야 합니다.
- 테스트는 명세서에 따라 수행되어야 하며, 주관적인 판단에 의존해서는 안 됩니다.

### 4. 조기에 테스트할수록 좋다

- 조기에 결함을 발견할수록 수정 비용이 적게 들고, 품질 향상에 기여할 수 있습니다.
- 요구사항 분석 단계부터 테스트 계획을 시작하는 것이 이상적입니다.

### 5. 테스트는 위험 기반으로 수행되어야 한다

- 모든 것을 테스트할 수 없으므로, 가장 중요한 기능과 위험 요소에 중점을 두어야 합니다.
- 위험 분석을 통해 테스트 우선순위를 결정합니다.

### 6. 테스트는 맥락에 따라 달라진다

- 은행, 항공, 의료 등 다양한 산업 분야에 따라 테스트 접근 방식이 달라져야 합니다.
- 시스템의 중요도에 따라 테스트 수준과 범위가 달라져야 합니다.

### 7. 오류 부재의 궤변 (Absence-of-errors fallacy)

- 결함이 없는 시스템이 사용자의 필요를 충족하지 못한다면 의미가 없습니다.
- 테스트는 사용자 요구사항과 시장 적합성도 고려해야 합니다.

## 테스트 프로세스

소프트웨어 테스트는 다음과 같은 5단계의 프로세스로 구성됩니다:

### 1. 계획 (Planning)

- 테스트 전략 수립
- 테스트 범위 정의
- 자원 및 일정 계획
- 테스트 도구 선택
- 역할 및 책임 정의

### 2. 분석 (Analysis)

- 테스트 대상 식별
- 테스트 조건 정의
- 위험 분석
- 테스트 데이터 요구사항 분석
- 테스트 환경 요구사항 분석

### 3. 설계 (Design)

- 테스트 케이스 설계
- 테스트 데이터 준비
- 테스트 시나리오 개발
- 테스트 절차 정의
- 테스트 스크립트 개발

### 4. 구현 (Implementation)

- 테스트 케이스 작성
- 테스트 환경 설정
- 테스트 자동화 스크립트 개발
- 테스트 데이터 생성
- 테스트 절차 문서화

### 5. 실행 (Execution)

- 테스트 실행
- 결함 기록 및 추적
- 테스트 결과 분석
- 결함 수정 확인
- 테스트 보고서 작성

## 테스트 심리학

### 개발자와 테스터의 사고방식 차이

**개발자 사고방식**:
- 시스템을 구축하고 동작하게 만드는 데 집중
- "성공"을 목표로 함
- 시스템이 작동하는 방식에 익숙

**테스터 사고방식**:
- 시스템이 실패하는 방식을 찾는 데 집중
- "실패"를 목표로 함
- 시스템의 취약점을 찾는 데 능숙

### 효과적인 커뮤니케이션 원칙

1. **비난이 아닌 협력**: 결함 보고는 비난이 아닌 개선의 도구
2. **객관적 표현**: 감정이 아닌 사실에 기반한 보고
3. **명확한 설명**: 결함 재현 단계를 명확히 설명
4. **건설적 피드백**: 개선 방향 제시
5. **공감과 존중**: 상대방의 입장을 이해하려는 노력

### 테스트 태도

- **의심의 태도**: "이건 작동할 거야"가 아니라 "이게 정말 작동할까?"라는 질문
- **분석적 사고**: 시스템을 다양한 각도에서 분석
- **주도적 태도**: 테스트 전략 수립과 실행에 주도적인 역할 수행
- **지속적 학습**: 새로운 테스트 기법과 도구에 대한 학습
