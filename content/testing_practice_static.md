# [기획/설계] 틈새 찾기

**"코딩 시작 전에 버그 절반은 잡을 수 있습니다."**

## 상세 내용

### 문서 검토의 중요성

- **요구사항 명세서 검토**: 모호한 표현, 불완전한 조건, 일관성 없는 정의를 사전에 식별
- **설계서 검토**: 설계 오류, 구현 상의 문제점, 테스트 가능성 등을 조기에 발견
- **코드 리뷰**: 코드 표준 준수, 보안 취약점, 성능 이슈 등을 실시간으로 확인

### 요구사항 명세서 검토

요구사항 명세서에서 주의 깊게 검토해야 할 항목들:
- 명확성: 표현이 모호하지 않은가?
- 완전성: 모든 요구사항이 포함되었는가?
- 일관성: 상충되는 요구사항은 없는가?
- 검증 가능성: 요구사항을 검증할 수 있는가?

### 설계서 검토

설계서 검토 시 중점적으로 확인해야 할 사항:
- 아키텍처 설계의 적절성
- 인터페이스 정의의 명확성
- 데이터 흐름의 정확성
- 오류 처리 방안

### 코드 리뷰

코드 리뷰의 핵심 포인트:
- 코딩 표준 준수 여부
- 보안 취약점 확인
- 성능 관련 문제
- 유지보수 용이성

### 실무 팁 - 정적 테스트

- **기획 단계부터 참여**: 테스터가 요구사항 분석 단계부터 참여하면 후속 테스트 효율성이 30% 이상 향상
- **체크리스트 활용**: 정형화된 검토 체크리스트를 통해 누락 방지
- **도구 활용**: 정적 분석 도구(SonarQube, ESLint 등)를 CI/CD에 통합

### 사례 - 정적 테스트

> 한 금융 앱 프로젝트에서 요구사항 명세서 검토를 통해 "계좌 이체 시 금액 0원 허용 여부"가 명확하지 않음을 발견. 구현 전에 명확화되어 추후 결함 3건 예방.
