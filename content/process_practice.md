## 테스트 계획 수립

### 1. 테스트 범위 정의

**템플릿:**
```markdown
## 테스트 범위

### 포함 (In Scope)
- [ ] 사용자 로그인/로그아웃
- [ ] 상품 검색 기능
- [ ] 장바구니 담기/삭제
- [ ] 결제 프로세스 (카드 결제만)

### 제외 (Out of Scope)
- [ ] 관리자 페이지 (다음 스프린트)
- [ ] 소셜 로그인 (개발 미완)
- [ ] 무통장 입금 (이번 릴리즈 제외)

### 리스크
| 기능 | 위험도 | 이유 | 대응 |
|-----|--------|------|------|
| 결제 | High | 금전 관련 | 집중 테스트, 실제 카드 테스트 |
| 검색 | Medium | 주요 기능 | 경계값, 특수문자 테스트 |
| UI | Low | 비즈니스 영향 적음 | 기본 테스트만 |
```

### 2. 리소스 배분

**테스터 2명 기준 (2주 스프린트):**
```
총 가용 시간: 2명 × 10일 × 6시간 = 120시간

배분:
- 테스트 계획 & 케이스 작성: 20시간 (17%)
- 기능 테스트: 40시간 (33%)
- 회귀 테스트: 25시간 (21%)
- 탐색적 테스트: 15시간 (13%)
- 버그 리포팅 & 재테스트: 15시간 (13%)
- 미팅 & 커뮤니케이션: 5시간 (4%)
```

### 3. 수용 기준 (Acceptance Criteria) 검증

**체크리스트:**
```
스토리: "사용자는 상품을 장바구니에 담을 수 있다"

수용 기준:
[ ] Given: 로그인된 상태
    When: 상품 상세 페이지에서 '장바구니 담기' 클릭
    Then: 장바구니에 상품이 추가되고 개수가 표시됨

[ ] Given: 비로그인 상태
    When: '장바구니 담기' 클릭
    Then: 로그인 페이지로 리다이렉트

[ ] Given: 재고 없는 상품
    When: '장바구니 담기' 클릭
    Then: "재고가 없습니다" 메시지 표시
```

## 회귀 테스트 전략

### 1. 회귀 테스트 범위 결정

**영향도 분석:**
```
변경 사항: 로그인 로직 수정

직접 영향:
- 로그인/로그아웃 기능 ✅ 테스트 필수

간접 영향:
- 세션 관리 ✅ 테스트 필수
- 권한 확인 ✅ 테스트 필수
- 사용자 정보 표시 ⚠️ 샘플 테스트

영향 없음:
- 상품 검색 ❌ 테스트 불필요
- 결제 ❌ 테스트 불필요 (세션은 확인)
```

### 2. 회귀 테스트 자동화

**자동화 우선순위:**
```
높음 (즉시 자동화):
✓ 반복적으로 실행
✓ 수동 테스트 시간 > 30분
✓ 데이터 입력 많음
✓ 복잡한 계산 검증

낮음 (수동 유지):
✗ 자주 변경되는 기능
✗ UI/UX 테스트
✗ 탐색적 테스트
✗ 일회성 테스트
```

**자동화 도구 선택:**

| 유형 | 도구 | 사용 시기 |
|-----|------|----------|
| **Web UI** | Selenium, Playwright, Cypress | 브라우저 자동화 |
| **Mobile** | Appium, XCUITest, Espresso | 앱 테스트 |
| **API** | Postman, REST Assured, pytest | API 검증 |
| **E2E** | Cypress, Playwright | 전체 플로우 |

```markdown
- 블로커: 2건

## 실행 상세

|-----|------|------|------|--------|
| 로그인 | 10 | 9 | 1 | 90% |
| 검색 | 15 | 15 | 0 | 100% |
| 장바구니 | 10 | 7 | 3 | 70% |
| 결제 | 10 | 7 | 3 | 70% |

## 발견 버그
| ID | 심각도 | 제목 | 상태 |
|----|--------|------|------|
| BUG-123 | Critical | 결제 실패 시 금액 차감 | Open |
| BUG-124 | High | 장바구니 수량 0 입력 가능 | Open |

1. BUG-123: 결제 로직 전체 테스트 불가능

## 내일 계획
- 블로커 해결 후 결제 모듈 재테스트
- 회귀 테스트 시작 (로그인, 검색)
```

### 2. 버그 수명주기 관리

```
[New] ──→ [Open] ──→ [In Progress] ──→ [Fixed] ──→ [Verified] ──→ [Closed]
           │                             │             │
           ↓                             ↓             ↓
        [Rejected]                  [Reopened]    [Reopened]
```

**각 단계 테스터 액션:**
- **New → Open**: 트리아지 미팅에서 우선순위 결정
- **Fixed → Verified**: 수정 사항 재테스트
- **Verified → Closed**: 배포 후 최종 확인
- **Reopened**: 재발 시 재오픈

### 3. 테스트 커버리지 모니터링

**측정 방법:**
```
테스트 커버리지 = (테스트된 요구사항 수 / 전체 요구사항 수) × 100

예시:
- 전체 요구사항: 50개
- 테스트 완료: 42개
- 커버리지: 84%
```

**목표:**
- 핵심 기능: 100%
- 주요 기능: 90% 이상
- 전체: 80% 이상

## 실무 팁

### 1. 효율적인 테스트 케이스 작성

**재사용 가능하게:**
```markdown
# 공통 전제 조건을 별도 관리
## PRE-001: 로그인된 상태

**전제 조건:** PRE-001 실행
```

### 2. 테스트 데이터 관리

**데이터 준비 전략:**

| 방법 | 장점 | 단점 | 사용 시기 |
|-----|------|------|----------|
| **수동 입력** | 간단 | 시간 소요 | 소규모 테스트 |
| **SQL 스크립트** | 빠름 | 유지보수 어려움 | DB 직접 접근 가능 |
| **API 호출** | 현실적 | 느림 | E2E 테스트 |
| **Mock 데이터** | 빠르고 격리 | 현실성 부족 | 단위 테스트 |

**예시: SQL 스크립트**
```sql
-- test_data_setup.sql
INSERT INTO users (email, password, name)
VALUES ('test@example.com', 'hashed_pw', '테스터');

INSERT INTO products (name, price, stock)
VALUES ('테스트 상품', 10000, 100);
```

### 3. 커뮤니케이션 스킬

**개발자와 협업:**
- ✅ DO: "API 응답 시 user_id가 null입니다" (사실)
- ❌ DON'T: "API가 이상해요" (모호)

**버그 우선순위 협상:**
```
상황: 개발자가 버그를 P3로 낮추려 함

테스터 대응:
"이해합니다. 하지만 이 버그는:
1. 일 사용자 1,000명 중 30%가 사용하는 기능
2. 결제 전환율에 직접 영향
3. 경쟁사 대비 UX 열위
따라서 P2 유지를 제안합니다."
```

**일일 스탠드업 발표:**
```
Bad: "어제 테스트했고, 오늘도 테스트합니다"

Good:
"어제: 로그인 모듈 10개 케이스 완료, 버그 2건 발견
오늘: 장바구니 모듈 15개 케이스 실행 예정

```

## 실전 사례

### 사례 1: 배포 직전 Critical 버그 발견

**상황:**
- 배포 1시간 전
- 결제 시 금액 계산 오류 발견

**대응:**
```
1. 즉시 PM, 개발 리더에게 에스컬레이션
2. 버그 재현 시나리오 공유 (5분 이내)
3. 비즈니스 영향 분석
   - 모든 결제 건 영향 (100%)
   - 금전적 손실 가능
4. 긴급 회의: 3가지 옵션 논의
   - A: 배포 연기 (선택)
   - B: 해당 기능만 롤백
   - C: 긴급 수정 (위험)
5. 결과: 배포 2일 연기, 수정 후 재테스트
```

**교훈:** 배포 전 최종 스모크 테스트는 필수

### 사례 2: 회귀 테스트 자동화로 70% 시간 절약

**Before:**
- 회귀 테스트: 수동 2일 (16시간)
- 매 스프린트 반복 → 월 64시간 소요

**After:**
```python
# Playwright로 자동화
@pytest.mark.regression
def test_login_logout_flow():
    page.goto("https://example.com")
    page.fill("#email", "test@example.com")
    page.fill("#password", "Test1234!")
    page.click("#login-button")
    assert page.locator("#user-name").inner_text() == "테스터"
    page.click("#logout-button")
    assert page.url == "https://example.com/login"
```

**결과:**
- 자동화: 30분
- 수동 검증: 4시간
- 총 시간 절감: 70%

### 사례 3: 탐색적 테스트로 Edge Case 발견

**상황:**
- 모든 테스트 케이스 통과
- 배포 후 사용자 문의 급증

**발견 버그:**
```
재현 시나리오:
1. 장바구니에 상품 A 담기
```

**원인:** 동시성 이슈 미고려

**개선:**
- 탐색적 테스트 시간 배분: 전체의 10-15%
- "만약 ~한다면?" 시나리오 작성

1. 테스트 범위 정의
