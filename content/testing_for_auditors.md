# 정보시스템 감리사가 현장에서 관심 가지는 테스팅

## 1. 정적 테스팅 (Static Testing): 문서 및 코드 검토

감리 현장에서 가장 먼저, 그리고 비중 있게 이루어지는 기법입니다.

### 산출물 검토
- 설계서, 요구사항 정의서, 테스트 결과서 등이 표준 가이드라인을 준수했는지 확인합니다.

### 코드 인스펙션(Inspection)
- 핵심 로직이나 보안 취약점이 우려되는 부분을 선별하여 소스 코드를 직접 눈으로 확인하거나 자동화 도구(검사 도구) 결과 리포트를 검토합니다.

### 추적성 확인
- 요구사항 정의서의 내용이 설계서와 소스 코드, 그리고 테스트 시나리오까지 누락 없이 이어졌는지(Traceability) 확인합니다.

## 2. 블랙박스 테스팅 (Black-box Testing): 사용자 관점

감리사는 내부 로직보다는 **'입력에 따른 출력'**이 비즈니스 요구사항과 일치하는지에 집중합니다.

### 경계값 분석 (Boundary Value Analysis)
- 가장 흔한 오류 지점인 입력 범위의 최소/최대값 근처를 찔러봅니다. (예: 1~100 입력창에 0이나 101을 입력)

### 동등 분할 (Equivalence Partitioning)
- 유효한 값과 유효하지 않은 값을 대표하는 데이터 몇 가지를 골라 시스템의 반응을 확인합니다.

### 오류 추정 (Error Guessing)
- 감리사의 경험과 직관을 사용하여 "이런 상황에서는 보통 에러가 나더라" 하는 케이스(예: 데이터 입력 중 강제 종료, 특수문자 입력 등)를 테스트합니다.

## 3. 비기능 테스팅 (Non-functional Testing): 품질 및 안정성

시스템이 단순히 돌아가는 것을 넘어, 잘 돌아가는지를 확인하는 단계입니다.

### 보안성 테스팅
- SQL Injection, Cross-Site Scripting(XSS) 등 주요 보안 약점이 조치되었는지 점검 도구 결과와 실제 칸투 시도를 통해 확인합니다.

### 성능/부하 테스팅 검토
- 실제 운영 환경과 유사한 부하를 주었을 때 응답 시간(Response Time)이 지연되지 않는지, 시스템이 다운되지 않는지 결과 리포트를 검토하고 필요시 직접 시연(Witness Test)을 참관합니다.

## 4. 재현 및 입증 테스팅 (Witness Testing)

감리사가 직접 모든 테스트를 수행하기보다는, 개발자나 운영자가 수행하는 테스트를 옆에서 참관하거나 중요 케이스를 재현하도록 요청합니다.

### 확인 테스팅
- 결함이 수정되었다고 보고된 항목들이 실제로 수정되었는지 다시 확인합니다.

### 회귀 테스팅 (Regression Testing)
- 특정 기능을 수정했을 때, 그와 연결된 다른 기존 기능들이 영향(Side Effect)을 받지 않았는지 샘플링하여 점검합니다.
