# 무작정 누르기(Monkey Test)와 탐색적 테스팅은 무엇이 다른가?

테스터들 사이에서 흔히 혼동하는 두 개념, **'몽키 테스팅(Monkey Testing)'**과 **'탐색적 테스팅(Exploratory Testing)'**의 차이점을 실무 관점에서 정리해 드립니다.

결론부터 말씀드리면, 몽키 테스팅은 '운'에 맡기는 것이고, 탐색적 테스팅은 테스터의 '지능'을 활용하는 것입니다.

## 📂 [테스트 수행] 무작정 누르기 vs 전략적으로 파고들기

### 1. 몽키 테스팅 (Monkey Testing): "예측 불투의 무작위 공격"

원숭이가 키보드를 두드리는 것처럼, 아무런 로직이나 계획 없이 시스템을 마구잡이로 조작하는 기법입니다.

- **수행 방식**: 화면 전체를 빠르게 연타하기, 말도 안 되는 긴 문자열 입력하기, 배터리 탈착 등 무작위 입력.

- **핵심 목적**: 시스템의 강건성(Robustness) 확인. 즉, "이렇게까지 하는데 안 죽고 버텨?"를 테스트합니다.

- **장점**: 테스터가 생각지 못한 지점에서 시스템 크래시(Crash)나 메모리 누수를 발견할 수 있습니다.

- **실무적 위치**: 주로 자동화 툴(Adb Monkey 등)을 이용해 밤새 돌려놓는 용도로 많이 쓰입니다.

### 2. 탐색적 테스팅 (Exploratory Testing): "경험에 기반한 추적"

테스트 설계와 수행을 동시에 진행하는 고도의 기법입니다. 시나리오(TC)에 갇히지 않되, 명확한 **목표(Charter)**를 가지고 움직입니다.

- **수행 방식**: "이 기능은 A와 B가 연결되어 있으니, A를 수정하고 B를 결제하면 에러가 날 것 같다"는 가설을 세우고 검증합니다.

- **핵심 목적**: 짧은 시간 안에 중요한 결함을 효율적으로 찾아내는 것.

- **장점**: 기획서의 빈틈이나 복잡한 비즈니스 로직 사이의 모순을 잡아내는 데 탁월합니다.

- **실무적 위치**: 기획서가 부실하거나, 배포 직전 시간이 촉박할 때 숙련된 테스터가 '사냥꾼'처럼 투입됩니다.

### 3. 한눈에 비교하는 차이점

| 구분 | 몽키 테스팅 (Monkey) | 탐색적 테스팅 (Exploratory) |
|------|----------------------|-----------------------------|
| 주체 | 주로 도구 (또는 초보자) | 숙련된 테스터 |
| 지적 능력 | 사용 안 함 (Random) | 적극 사용 (Cognitive) |
| 계획성 | 없음 | **차터(Charter)**라는 느슨한 가이드 존재 |
| 재현성 | 매우 낮음 (뭘 눌렀는지 모름) | 높음 (사고의 흐름을 기록함) |
| 주요 발견 | 시스템 크래시, 하드웨어 에러 | 로직 오류, UX 미비, 예외 상황 결함 |

### 4. 실무 팁: 탐색적 테스팅을 '제대로' 하는 법

무작정 누르는 몽키 테스팅이 되지 않으려면 **'세션(Session)'**과 **'기록'**이 필요합니다.

- **차터(Charter) 정의**: "쿠폰 만료 직전 결제 시의 예외 상황을 탐색한다"와 같은 구체적 목표 설정.

- **시간 제한(Time-boxing)**: 60분~90분 정도 집중해서 수행.

- **테스트 로그(Log) 작성**: 어떤 경로로 이동했고, 어떤 가설을 세웠는지 간략히 메모. (그래야 버그 발견 시 재현이 가능합니다.)

## 📝 실무 적용 예시 (장바구니 테스트 시)

- **몽키 테스터**: 장바구니 버튼을 초당 10번 연타하거나, 휴대폰 비행기 모드를 켰다 껐다 하며 앱이 죽는지 봅니다.

- **탐색적 테스터**: "장바구니에 상품을 담은 상태에서 다른 브라우저로 로그인해 해당 상품을 삭제하면, 기존 브라우저 결제 시 어떻게 될까?"라는 시나리오를 즉석에서 설계해 수행합니다.
