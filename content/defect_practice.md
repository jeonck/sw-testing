## 왜 결함 보고서가 중요한가?

좋은 결함 보고서는:
- 개발자 수정 시간 **60% 단축**
- 불필요한 커뮤니케이션 **70% 감소**
- 결함 재발견율 **50% 감소**

나쁜 보고서는 시간을 낭비하고 팀 관계를 악화시킵니다.

## 결함 보고서 필수 항목

### 기본 구성

| 항목 | 설명 | 예시 |
|-----|------|------|
| **제목** | 한 줄 요약 (What + Where) | "[로그인] 비밀번호 찾기 링크 클릭 시 404 에러" |
| **심각도** | Critical / High / Medium / Low | High |
| **우선순위** | P1 (즉시) / P2 (긴급) / P3 (보통) / P4 (낮음) | P2 |
| **환경** | OS, 브라우저, 버전 등 | Chrome 120, Windows 11 |
| **재현율** | 항상 / 간헐적 (몇 %) | 항상 (100%) |
| **발견 일시** | 날짜 및 시간 | 2025-01-15 14:30 |

## 심각도 & 우선순위 가이드

### 심각도 (Severity) - 영향 범위

| 레벨 | 기준 | 예시 |
|-----|------|------|
| **Critical** | 시스템 중단, 데이터 손실 | 결제 시스템 다운, DB 데이터 삭제 |
| **High** | 주요 기능 불가 | 로그인 실패, 검색 안됨 |
| **Medium** | 기능 동작하나 비정상 | 잘못된 계산 결과, UI 깨짐 |
| **Low** | 사소한 문제 | 오타, 정렬 오류 |

### 우선순위 (Priority) - 수정 시급성

| 레벨 | 기준 | 예시 |
|-----|------|------|
| **P1** | 즉시 수정 (Hot Fix) | 운영 서버 장애 |
| **P2** | 다음 배포 전 필수 | 핵심 기능 버그 |
| **P3** | 가능하면 이번 스프린트 | 부가 기능 개선 |
| **P4** | 다음 기회에 | 편의성 개선 |

**심각도 vs 우선순위 차이:**
- 높은 심각도 + 낮은 우선순위: 회원 탈퇴 기능 버그 (중요하지만 사용 빈도 낮음)
- 낮은 심각도 + 높은 우선순위: 메인 페이지 오타 (사소하지만 모든 사용자가 봄)

## 재현 절차 작성법

### ❌ 나쁜 예시
```
버튼을 클릭하면 에러가 발생합니다.
```

### ✅ 좋은 예시
```
**재현 절차:**
1. https://example.com 접속
2. 상단 '로그인' 버튼 클릭
3. 이메일: test@example.com 입력
4. 비밀번호: Test1234! 입력
5. '비밀번호 찾기' 링크 클릭

**기대 결과:**
- 비밀번호 재설정 페이지로 이동

**실제 결과:**
- 404 Not Found 에러 페이지 표시
- URL: https://example.com/reset-password (존재하지 않음)

**재현율:** 100% (5회 시도 / 5회 재현)
```

### 재현 절차 작성 5원칙

1. **번호 매기기**: 단계마다 숫자 부여
2. **구체적으로**: "적절히 입력" ❌ → "test@example.com 입력" ✅
3. **순서대로**: 실제 수행한 순서 그대로
4. **전제 조건 명시**: 로그인 필요 여부, 데이터 준비사항 등
5. **재현율 명시**: 항상 / 50% / 10회 중 3회 등

## 효과적인 증거 첨부

### 1. 스크린샷

**필수 포함 사항:**
- 전체 화면 (툴바, URL 포함)
- 에러 메시지 전문
- 타임스탬프 (가능하면)

**도구 추천:**
- Windows: `Win + Shift + S`
- Mac: `Cmd + Shift + 4`
- Chrome DevTools: F12 → Console 탭 캡처

### 2. 동영상

**언제 필요한가:**
- 재현 절차가 복잡할 때
- 애니메이션/타이밍 이슈
- 간헐적 버그

**도구 추천:**
- [Loom](https://www.loom.com) - 화면 녹화 + 설명
- [ScreenToGif](https://www.screentogif.com) - GIF 제작
- OBS Studio - 무료 고품질 녹화

### 3. 로그 파일

**어떤 로그를 첨부할까:**
```
✓ 브라우저 Console 로그 (F12)
✓ Network 탭 (API 실패 시)
✓ 서버 로그 (해당 시간대)
✓ 모바일 로그 (Android: Logcat, iOS: Console)
```

**로그 추출 예시:**
```bash
# Chrome DevTools Console
우클릭 → Save as...

# 특정 시간대 서버 로그
grep "2025-01-15 14:3" /var/log/app.log
```

## 결함 보고서 템플릿

### 템플릿 1: 기본 형식

```markdown
## BUG-1234: [모듈명] 한 줄 요약

**환경:**
- OS: Windows 11
- 브라우저: Chrome 120.0.6099.129
- 디바이스: Desktop
- 빌드 버전: v2.3.1

**심각도:** High
**우선순위:** P2
**재현율:** 100%

**전제 조건:**
- 로그인된 상태
- 장바구니에 상품 1개 이상

**재현 절차:**
1. 장바구니 페이지 접속
2. 수량을 '0'으로 변경
3. '수량 업데이트' 버튼 클릭

**기대 결과:**
- "수량은 1개 이상이어야 합니다" 에러 메시지 표시
- 수량이 기존 값으로 유지

**실제 결과:**
- 상품이 장바구니에서 삭제됨
- 사용자에게 확인 없이 즉시 삭제

**첨부:**
- 스크린샷: screenshot_001.png
- Console 로그: console_error.txt

**추가 정보:**
- Safari, Firefox에서도 동일하게 재현됨
- 모바일 앱에서는 정상 동작
```

### 템플릿 2: 간헐적 버그

```markdown
## BUG-1235: [결제] 간헐적 결제 실패

**재현율:** 30% (10회 시도 중 3회 발생)

**발생 패턴:**
- 주로 오후 2~4시 사이 발생
- 결제 금액과 무관
- 특정 브라우저와 무관

**재현 시도:**
- 시도 1: 성공
- 시도 2: 성공
- 시도 3: **실패** ← 타임아웃 에러
- 시도 4: 성공
- 시도 5: **실패** ← 네트워크 에러

**추정 원인:**
- 서버 부하 관련 추정 (피크 시간대 일치)
- Network 탭에서 응답 시간 10초 이상 확인

**첨부:**
- 발생 시간대 로그: error_times.xlsx
- Network 캡처: network_timeout.har
```

## 커뮤니케이션 베스트 프랙티스

### 1. 사실 vs 추측 구분

| ❌ 나쁜 예시 | ✅ 좋은 예시 |
|------------|------------|
| "DB가 잘못됐어요" | "결제 완료 후 주문 테이블에 데이터 없음 확인" |
| "서버가 느려요" | "API 응답 시간 5.3초 측정 (기준: 3초 이내)" |
| "코드가 이상해요" | "동일 API 호출이 2번 발생 (Network 탭 확인)" |

### 2. 감정 제거, 사실만

| ❌ 피해야 할 표현 | ✅ 사용해야 할 표현 |
|----------------|------------------|
| "너무 이상해요" | "예상과 다르게 동작합니다" |
| "전혀 안 돼요" | "다음 시나리오에서 실패합니다" |
| "또 똑같은 버그" | "v2.1에서 수정된 이슈와 유사합니다 (BUG-1100 참조)" |

### 3. 비즈니스 영향 명시

```markdown
**비즈니스 영향:**
- 예상 영향 고객: 일 평균 로그인 사용자 1,000명
- 손실 추정: 결제 전환율 5% 감소 (일 매출 -50만원)
- 고객 문의: 지난 3일간 15건 접수
```

## 실무 팁

### 1. 결함 보고서 체크리스트

재제출하기 전 확인:
```
[ ] 제목이 명확한가? (무엇이 + 어디서)
[ ] 재현 절차가 5단계 이내인가?
[ ] 스크린샷/로그를 첨부했는가?
[ ] 환경 정보가 완전한가?
[ ] 재현율을 명시했는가?
[ ] 기대 결과 vs 실제 결과를 비교했는가?
[ ] 비슷한 버그가 이미 등록되지 않았나?
```

### 2. 도구 활용

| 도구 | 용도 | 추천 이유 |
|-----|------|----------|
| **Jira** | 이슈 트래킹 | 업계 표준, 강력한 워크플로우 |
| **GitHub Issues** | 오픈소스 프로젝트 | 개발자 친화적 |
| **Bugzilla** | 엔터프라이즈 | 커스터마이징 자유도 높음 |
| **Notion** | 소규모 팀 | 유연한 템플릿, 협업 용이 |

### 3. 자주 하는 실수

| 실수 | 결과 | 해결 |
|-----|------|------|
| 재현 절차 생략 | 개발자가 재현 못함 | 항상 단계별 작성 |
| 환경 정보 누락 | "제 PC에서는 안 돼요" | OS, 브라우저, 버전 명시 |
| 여러 버그 혼합 | 일부만 수정되고 종료 | 1개 리포트 = 1개 버그 |
| 모호한 제목 | 검색/분류 어려움 | [모듈] + 구체적 현상 |

## 실전 사례

### 사례 1: 간결한 보고서로 60% 시간 단축

**Before (나쁜 예시):**
> "로그인이 안 됩니다. 확인 부탁드립니다."

**After (좋은 예시):**
> **[로그인] 특수문자 포함 비밀번호 입력 시 400 에러**
>
> 재현 절차:
> 1. 로그인 페이지 접속
> 2. 이메일: test@example.com
> 3. 비밀번호: Pass@123! (특수문자 포함)
> 4. 로그인 버튼 클릭
>
> 실제 결과: "Invalid request" 에러
> Console: `SyntaxError: Unexpected token @`

**효과:** 개발자가 즉시 원인 파악 (비밀번호 특수문자 이스케이프 누락)

### 사례 2: 비즈니스 영향 강조로 우선순위 상향

**Before:**
> "검색 결과가 이상합니다"

**After:**
> **[검색] 인기 검색어 '아이폰'이 결과 0건**
>
> **비즈니스 영향:**
> - '아이폰' 월 검색량: 10,000회
> - 예상 전환율 손실: 3% (300건)
> - 경쟁사 대비 검색 품질 하락
>
> **재현:** (생략)

**결과:** P3 → P1 상향, 당일 Hot Fix 배포

### 사례 3: 간헐적 버그 패턴 분석

**상황:** "가끔 결제가 안 돼요" 보고 접수

**조사:**
- 발생 시간대 기록: 14:00~16:00 집중
- 공통점 분석: 모두 카드 결제, 금액 무관
- 서버 로그 확인: CPU 사용률 95% 이상일 때 타임아웃

**보고서:**
> **[결제] 피크 시간대 카드 결제 타임아웃 (재현율 30%)**
>
> **발생 패턴:**
> - 시간: 평일 14:00~16:00
> - CPU 사용률 95% 이상일 때
> - 결제 API 응답 시간 10초 초과 → 타임아웃
>
> **제안:**
> - 결제 API 타임아웃 10초 → 30초 증설
> - 또는 서버 Auto Scaling 고려

**결과:** 서버 증설로 해결, 결제 성공률 30% 향상

**우선순위:** P1

- 로그인된 상태
- 정상 재고가 있는 상품

**재현 절차:**
1. 홈페이지 접속
2. 임의의 상품 선택

**실제 결과:**
- '결제하기' 버튼 비활성화 (회색)
- 클릭 불가

**추가 정보:**
- Console에 에러 없음
- Network 탭 정상
- 다른 브라우저(Firefox)에서도 동일

**첨부:**
- 스크린샷: cart_button_disabled.png
