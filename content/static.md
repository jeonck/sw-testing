# 정적 테스트 (Static Testing)

## 정적 기법의 개요

정적 테스트는 코드를 실행하지 않고 소프트웨어의 결함을 찾는 기법입니다. 이는 코드 검토, 문서 검토, 정적 분석 도구 사용 등을 포함합니다.

### 정적 테스트의 이점

- **조기 결함 발견**: 코드 실행 전에 결함을 발견
- **비용 절감**: 초기 단계에서 결함 수정 비용이 적음
- **보안 강화**: 보안 취약점 조기 발견
- **코드 품질 향상**: 코딩 표준 준수 확인
- **지식 공유**: 팀원 간 코드 이해도 향상

### 정적 테스트와 동적 테스트의 비교

| 구분 | 정적 테스트 | 동적 테스트 |
|------|-------------|-------------|
| 실행 여부 | 코드 실행 없음 | 코드 실행 필요 |
| 발견 가능한 결함 | 논리 오류, 스타일 문제, 구조적 문제 | 런타임 오류, 성능 문제 |
| 수행 시기 | 개발 중, 코드 리뷰 시 | 빌드 후, 테스트 환경에서 |
| 도구 활용 | 정적 분석 도구 | 테스트 자동화 도구 |

## 리뷰 프로세스

### 리뷰의 유형

#### 1. 비공식 리뷰 (Informal Review)

- **정의**: 간단한 코드 검토 또는 동료 간 검토
- **형식**: 페어 프로그래밍, 코드 리뷰 요청
- **장점**: 빠르고 유연한 피드백
- **단점**: 공식적인 추적 어려움

#### 2. 워크스루 (Walkthrough)

- **정의**: 개발자가 자신의 코드를 다른 팀원에게 설명
- **목표**: 코드 이해도 향상, 간단한 결함 발견
- **진행자**: 코드 작성자
- **참여자**: 팀원, 테스터, PM 등

#### 3. 기술 리뷰 (Technical Review)

- **정의**: 기술 전문가들이 기술적 측면 검토
- **목표**: 기술적 정확성, 표준 준수 확인
- **진행자**: 기술 전문가
- **문서화**: 공식적인 문서화 필요

#### 4. 관리 리뷰 (Management Review)

- **정의**: 관리자 수준에서 진행하는 리뷰
- **목표**: 프로젝트 진행 상황, 자원 배분, 일정 등 검토
- **참여자**: 관리자, PM, 주요 이해관계자

#### 5. 인스펙션 (Inspection)

- **정의**: 체계적이고 공식적인 리뷰 프로세스
- **역할**:
  - **리더**: 리뷰 프로세스 주도
  - **작성자**: 검토 대상의 원저자
  - **리뷰어**: 결함 탐지 및 보고
  - **사서**: 결함 및 리뷰 기록 관리
  - **중재자**: 논쟁 조정

### 리뷰 프로세스 단계

1. **계획 (Planning)**
   - 리뷰 목표 설정
   - 참여자 선정
   - 일정 수립
   - 리뷰 자료 준비

2. **준비 (Preparation)**
   - 리뷰 자료 분배
   - 참여자 준비 시간 확보
   - 리뷰 체크리스트 제공

3. **리뷰 (Review/Meeting)**
   - 리뷰 회의 진행
   - 결함 식별
   - 토론 및 기록

4. **수정 (Rework)**
   - 식별된 결함 수정
   - 수정 사항 문서화

5. **추적 (Follow-up)**
   - 수정 사항 검증
   - 리뷰 결과 보고
   - 품질 지표 수집

## 정적 분석 도구

### 복잡도 분석 도구

#### Cyclomatic Complexity (McCabe 복잡도)
- **정의**: 프로그램의 독립된 경로 수를 측정
- **계산 방법**: 결정 수 + 1
- **기준**:
  - 1-10: 단순
  - 11-20: 복잡
  - 21-50: 매우 복잡
  - 50+: 비정상적으로 복잡

#### 도구 예시
- **SonarQube**: 코드 품질 및 보안 분석
- **ESLint**: JavaScript/TypeScript 코드 분석
- **SonarLint**: IDE 내부 정적 분석
- **Checkstyle**: Java 코딩 표준 검사
- **PMD**: Java, JavaScript, PL/SQL 등 분석
- **FindBugs/SpotBugs**: Java 결함 탐지

### 코딩 표준 준수 확인

#### 일반적인 코딩 표준
- 변수명 명명 규칙
- 주석 표준
- 함수 길이 제한
- 중복 코드 식별
- 보안 취약점 검사

#### 도구별 특징
- **SonarQube**: 다양한 언어 지원, 보안 취약점 탐지
- **ESLint**: JavaScript/TypeScript 중심, 플러그인 확장성
- **Checkstyle**: Java 중심, 규칙 커스터마이징 용이
- **PMD**: 코드 복잡도, 성능, 보안 이슈 탐지

### 정적 분석 도구 적용 전략

#### CI/CD 파이프라인 통합
```
코드 커밋 → 정적 분석 도구 실행 → 품질 게이트 통과 여부 확인 → 다음 단계 진행
```

#### 품질 게이트 설정
- 코드 커버리지 기준
- 복잡도 임계값
- 결함 밀도 기준
- 중대 오류 발생 여부

#### 리포트 및 추적
- 정적 분석 결과 리포트 생성
- 추적 가능성 확보
- 추세 분석

## 정적 테스트의 최적화

### 정적 테스트 적용 시 고려사항

1. **도구 선택**
   - 프로젝트 언어 및 기술 스택
   - 팀의 역량과 익숙도
   - 유지보수 용이성
   - 통합 가능성

2. **프로세스 설계**
   - 리뷰 빈도
   - 참여자 역할 정의
   - 결함 처리 절차
   - 측정 지표 설정

3. **문화적 요소**
   - 비난이 아닌 개선 중심
   - 개발자와 테스터 협업
   - 지속적인 개선 문화

### 정적 테스트의 한계

- **런타임 결함 감지 불가**: 동적 테스트 필요
- **도구의 한계**: 모든 결함 탐지 불가
- **오진 가능성**: 거짓 양성 결과
- **설정 및 유지보수 비용**: 도구 설정 및 유지 관리 필요

정적 테스트는 소프트웨어 품질 향상을 위한 중요한 기법으로, 정적 분석 도구와 리뷰 프로세스를 효과적으로 결합하여 사용하는 것이 중요합니다. 이는 동적 테스트와 함께 사용될 때 가장 효과적인 품질 보증 체계를 구성할 수 있습니다.
